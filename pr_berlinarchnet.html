Berlin Architectural Network

Berlin Architectural Network is a Ruby on Rails Application built for the homonymous Meetup group, which I've created in 2017. Since then, it has attracted hundreds of architecture lovers who come together to support each other through collaboration and networked learning.

Together with my team from the Bootcamp at Le Wagon, we've built this web application from scratch with Ruby, HTML, SCSS and JavaScript.

Building a social platform as the final project in only 2 week's time, required a good understanding of data base architecture, overall enjoyable user experience as well as skills on time and product management.

Goal and Process

The purpose of this project is to support architects, who are new Berlin, on getting into the city's architecture scene and give the Meetup group an online presence. The design process had 3 major phases:

Phase 1 - Sign up

After defining each step of the user journey along with a Figma prototype, we designed the database schema and built the corresponding models on Rails. At this point we introduced Devise for authentication and Pundit for authorization, so that the user could sign up and become a member of the community.

Phase 2 - Browse

On the second phase the user should be able to visit an office's page and check out a member's activity. For the database seeding, we used the JSON format due to it fast execution. We also worked with services such as Cloudinary to host and upload images and Mapbox to display the location of the offices.

Phase 3 - Interact

The last phase enabled the user to RSVP to an event, review it and chat with a member. AJAX was used for rspving and posting comments, which allowed new content to be displayed without refreshing. For the chat, we've built a bidirectional Websocket connection combined with Action Cable for real-time messaging.

Challenges and solving process

The process was not always smooth and we had some bumps on the way that helped us grow as junior developers.

Database Architecture

The database was a highlight of this project because of its complexity. There was a total of 12 models.

Something unusual about this schema is the comments table that allows foreign keys to be nullified. This decision was made in order to reduce the number of tables. Even though this is valid and works, it's something i would rather break into smaller tables in the future, for the sake of simplicity.

It was nonetheless an opportunity to try and see a different solution on action.

Teamwork on Github

By working on someone else's feature, i've caused conflicts on github, which had to be reviewed line by line and took us time. After that we've started to practice pair programming.

A versioning issue would not allow me to push my code to the repository, so I've also learned how to remove empty files and fix the head manually on gibhub on the night prior to the demo!

Agile Mindset

As an architect, i've explored the benefits of agile management while coordinating small teams. This experience guided me throughout the process of building this app.

We've implemented scrum rituals such as daily standups as well as a kanban board on trello, which helped us visualize our progress and workflow as a team. Despite the timeline of 2 weeks, we were able to put the project up and running for the demo and still had time to work on one extra feature.
